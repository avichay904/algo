# בחירת פעילויות (Activity Selection)

## 🌸 אינטואיציה

נתונות n פעילויות, כל פעילות i עם **זמן התחלה `s[i]`** ו**זמן סיום `f[i]`**. מחפשים קבוצה גדולה ביותר של פעילויות **לא-חופפות** (שיהיו כמה שיותר פעילויות, לא מינימום זמן).

**האסטרטגיה החמדנית**: תמיד לבחור את הפעילות עם **זמן הסיום הכי מוקדם** מבין הפעילויות שעדיין אפשריות.

זה אופטימלי! אינטואיציה: פעילות שמסתיימת מוקדם משאירה הכי הרבה זמן פנוי לשאר.

## 📐 פסאודו-קוד

הנחה: הפעילויות ממוינות לפי `f[i]` בסדר עולה.

```
ACTIVITY-SELECTOR(s, f):
  n = length(s)
  A = {a₁}                      # הפעילות הראשונה תמיד נבחרת
  last = 1

  for i = 2 to n:
    if s[i] >= f[last]:          # הפעילות i לא חופפת לאחרונה שנבחרה
      A = A ∪ {aᵢ}
      last = i

  return A
```

גרסה רקורסיבית:
```
RECURSIVE-ACTIVITY-SELECTOR(s, f, k, n):
  m = k+1
  while m <= n and s[m] < f[k]:
    m = m+1
  if m <= n:
    return {aₘ} ∪ RECURSIVE-ACTIVITY-SELECTOR(s, f, m, n)
  else:
    return ∅
```

## ⏱️ סיבוכיות

- **זמן:** `Θ(n log n)` אם צריך למיין, או `Θ(n)` אם כבר ממויין לפי `f`
- **זיכרון:** `Θ(1)` נוסף (רק מצביע `last`)

## 💡 טיפ למבחן

- **המיון הוא לפי `f[i]` (זמן סיום), לא לפי `s[i]`!** טעות נפוצה: למיין לפי זמן התחלה.
- **הוכחת נכונות בטענת ההחלפה**: נניח יש פתרון אופטימלי `O` שאינו כולל את `a₁` (הפעילות עם ה-f הקטן ביותר). אז אפשר להחליף את הפעילות הראשונה ב-`O` ב-`a₁` וזה עדיין פתרון תקין, עם אותו מספר פעילויות.
- **שונה מ-Job Scheduling** (תזמון משימות עם deadlines ועונשים) — שם האסטרטגיה לפי זמן מוגבל/עונש.
- **לא עובד גרידי** אם המטרה היא "מזמן כולל מקסימלי" (אז זו בעיית אריזה אחרת — Weighted Job Scheduling, DP).
