# קידוד האפמן (Huffman Coding)

## 🌸 אינטואיציה

בקידוד קלאסי (ASCII), כל תו תופס מספר קבוע של ביטים (למשל 8). בקידוד האפמן, **תווים נפוצים מקבלים קוד קצר, תווים נדירים מקבלים קוד ארוך**.

התוצאה: אם בטקסט יש הרבה 'a' ומעט 'z', המהלך יהיה יעיל בהרבה.

**תכונה חשובה**: קוד האפמן הוא **קוד ללא קידומת (prefix-free)** — אף קוד של תו אינו קידומת של קוד של תו אחר. זה מאפשר פענוח חד-משמעי ללא מפרידים.

## 📐 פסאודו-קוד

```
HUFFMAN(C):
  # C = קבוצה של n תווים, לכל תו c יש תדירות c.freq
  n = |C|
  Q = min-priority-queue(C)    # לפי freq

  for i = 1 to n-1:
    z = new node
    z.left  = x = EXTRACT-MIN(Q)
    z.right = y = EXTRACT-MIN(Q)
    z.freq  = x.freq + y.freq
    INSERT(Q, z)

  return EXTRACT-MIN(Q)        # השורש של העץ
```

קריאת הקוד של כל תו: מסלול מהשורש לעלה — שמאל = 0, ימין = 1.

## ⏱️ סיבוכיות

- **זמן:** `O(n log n)` — כל פעולת min-heap היא `O(log n)` ויש `2(n-1)` פעולות
- **זיכרון:** `O(n)`

## 💡 טיפ למבחן

- **חמדני**: בכל שלב מאחדים את שני הצמתים עם **הכי קטן freq**. זה תמיד אופטימלי.
- **הוכחת נכונות** — טענת ההחלפה: אם שני התווים הכי פחות תדירים לא אחים בעץ האופטימלי, אפשר להחליף בלי להגדיל את עלות הקוד.
- **אורך ממוצע** של קוד האפמן חסום בין `H(X)` ל-`H(X)+1` כאשר H היא אנטרופיה של שאנון.
- שאלה קלאסית: "מה אם כל התדירויות שוות?" — התשובה: האפמן יעשה עץ מאוזן, וכל הקודים יהיו באותו אורך `⌈log n⌉`.

**דוגמה מה-PDF**: תדירויות `a=3, b=1, c=1, d=4`. סדר איחוד: {b,c}→2, {a,2}→5, {d,5}→9. סה"כ חסכון לעומת קידוד אחיד.
